# GitHub에서만 작업하는 사용자용: bot 정상상태 반영 가이드

이 문서는 **로컬 개발 없이 GitHub UI만 사용**하는 기준으로 작성되었습니다.

## 목적

- 지금 정상 작동하는 `bot` 상태를 저장소에 안전하게 반영
- 다음 업데이트 때 같은 장애를 줄이기

## 전제

- 이 저장소는 `bot`(텔레그램 사용자용)과 `automation`(백엔드 자동화)이 분리되어 있음
- `bot`만 수정했으면 `bot` 서비스만 배포 대상으로 확인해야 함

---

## 1) PR 머지 전에 꼭 확인할 것

GitHub PR 화면에서 아래만 확인하세요.

1. 변경 파일에 `bot/` 경로만 있는지
2. CI(체크) 결과가 초록색인지
3. PR 설명에 "왜 바꿨는지 / 무엇을 바꿨는지 / 테스트 결과"가 있는지

---

## 2) 머지 후 Railway(또는 배포 플랫폼)에서 확인할 것

1. 배포 대상 서비스가 `bot`인지 확인
2. 환경변수 `BOT_TOKEN`이 설정되어 있는지 확인
3. **채널 자동 포스팅(VIP 프리미엄 발송)** 사용 시: `POSTS_DB_PATH`(공유 posts.db 경로), `ADMIN_ID`, `PROMO_PAGE_URL`, `GAME_PAGE_URL`, `PROMO_CODE`, `GEMINI_API_KEY`(선택) 설정. `/admin` 메뉴에서 발송 가능.
4. 배포 완료 후 Telegram에서 아래 순서로 실동작 테스트
   - `/start`
   - 언어 선택 버튼
   - 메뉴 선택
   - `close` 버튼

이 4개가 정상 동작하면 반영 성공입니다.

---

## 3) 다음 업데이트 때 안전하게 하는 규칙

1. 한 PR에서 큰 변경을 한 번에 하지 않기
2. `bot` 변경과 `automation` 변경을 같은 PR에 섞지 않기
3. 장애 발생 시 "기록 삭제"보다 "되돌리는 PR"(revert)을 우선 사용

---

## 4) 문제 발생 시 빠른 복구 원칙

1. 마지막 정상 PR 번호를 기록해 둠
2. 장애 PR은 즉시 되돌리는 PR 생성
3. 복구 후 원인 분석 PR은 별도로 분리

이렇게 하면 서비스 중단 시간을 가장 짧게 가져갈 수 있습니다.
